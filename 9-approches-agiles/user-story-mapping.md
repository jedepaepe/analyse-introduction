# User story mapping

Jeff Patton définit le user story mapping comme
*Talk about the user’s journey through your product by building a simple model that tells your user’s story as you do; ref : [User Story Mapping](https://jpattonassociates.com/story-mapping/)*.
Ensuite il précise :
*More importantly it’ll keep your users and what they’re doing with your product front and center in your products. That’s better than getting lost in feature arguments – like what normally happens in software development.*

Le user story mapping est un récit narratif du parcours de l'utilisateur à travers l'application :
- Il met le focus sur le flux d'activités de l'utilisateur et la valeur ajoutée qu'il en tire : Quel est le but de l'utilisateur et que fait-il pour atteindre ce but ?
- Il permet d'organiser les fonctionnalités, de les ordonnés chronologiquement et par priorité.
- Il facile la définition de releases qui seront déployées en production.

Le story mapping (synonyme) définit les concepts suivants :
- Backbone (colonne vertébrale) : est l'ensemble des activités/
- Activité : ensemble de tâches, elles sont ordonnées suivant la narration, c'est-à-dire plus ou moins temporellement.
- User task (tâche utilisateur) : la tâche est l'objectif de l'utilisateur. La tâche est métier, elle ne dépend pas de l'application. Par exemple : choisir un mode de livraison, faire une demande de congé. Les user tasks sont décomposées en user stories. 
- User story (récit utilisateur) : décrit une façon de faire la tâche. Parfois, il n'y a qu'une US pour un tâche. Parois il y en a plusieurs. Par exemple : Je choisis le mode de livraison relais. Je choisis le mode de livraison à la maison. 

Voici comment Jeff Patton définit le backbone :
*The Backbone Organizes the Map
At the top of the map is the backbone, which sometimes has a couple
of different levels. You might start with the basic flow of the story,
which is one level. But, when it gets really long, it’s useful to go up one
more level to summarize things further.*

*When you read the backbone of the map from left to right, it tells a
story about all the people who use the system and what they do in
order to create and manage sites and content. The left-to-right order
is what I call a narrative flow, which is an academic way of saying the
order in which we’d tell the story. Of course, all these people are doing
everything all at once, and sometimes things don’t move in a perfect
order, but we know that. We just put them in an order that helps us
tell the story.*
*Ref : [livre User Story Mapping de Jeff Patton](https://jpattonassociates.com/story-mapping/)

L'idée est donc de chercher ce que fait l'utilisateur,
de les organiser sur une ligne temporelle imaginaire,
de les grouper par activité.
Le tout forme le support pour une narration de ce que fait l'utilisateur avec l'application.
